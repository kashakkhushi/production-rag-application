# Hybrid Retrieval Pipeline

The Enterprise Knowledge Assistant implements a robust **Hybrid Search and Reranking Pipeline** located entirely in `backend/app/rag/pipeline.py`. 

This approach mitigates the common flaws of naive semantic search (failing on exact keyword matches like names/acronyms) while retaining dense contextual understanding.

## Pipeline Steps

### 1. Dual Retrieval
For every incoming query, the system simultaneously queries two vector spaces:
- **Dense Retrieval (ChromaDB):** Fetches the top-K semantically similar chunks using embeddings generated by `all-MiniLM-L6-v2`. It handles conceptual similarity well.
- **Sparse Retrieval (BM25):** Fetches the top-K chunks based on exact lexical matches using the `rank-bm25` Okapi implementation.

### 2. Score Normalization
Since BM25 scores (unbounded) and Dense Cosine scores (0 to 1) exist on completely different scales, we must normalize them before combination.
The pipeline extracts the `min` and `max` scores for both dense and sparse results independently and applies **Min-Max Normalization** to compress all scores into a standardized `[0.0, 1.0]` range.

### 3. Weighted Combination & Deduplication
The normalized scores are then merged for each unique `chunk_id`. We apply the following configurable weights (defined in `.env`):
- **Dense Weight:** `0.6`
- **Sparse Weight:** `0.4`

*Why this ratio?* Semantic context generally proves slightly more reliable for natural language Q&A, but the 40% lexical weight prevents catastrophic forgetting of domain-specific keywords. The deduplicated candidates are sorted by their new combined score.

### 4. Cross-Encoder Reranking
The top combined candidates are passed to a Cross-Encoder (`ms-marco-MiniLM-L-6-v2`). 
Unlike dual-encoders (which pre-compute embeddings), a cross-encoder scores the query and candidate chunk *together*, capturing deep token-level interactions. The chunks are reranked based on this final cross-encoder score.

### 5. Hallucination Prevention (Thresholding)
Before generating an answer, `generate_answer()` in `generation.py` inspects the maximum reranker score. 
If the top candidate's confidence is below `MIN_RELEVANCE_THRESHOLD` (set to `0.3`), the pipeline aborts generation and falls back to a deterministic refusal: *"I could not find sufficient information in the uploaded documents."* This guarantees strictly grounded answers.
